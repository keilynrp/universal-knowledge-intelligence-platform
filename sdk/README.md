# UKIP SDK

Generated client artifacts for the UKIP API.

## What is here today

| Path | Status |
|---|---|
| `openapi.json` | ✅ Committed baseline, regenerated by `scripts/generate-sdk.mjs`, drift-gated in CI |
| `typescript/` | ⏳ Not yet generated — see [Generating the clients](#generating-the-clients) |
| `python/` | ⏳ Not yet generated |

The **contract** is stable and gated. The **clients** are the next slice.

## Regenerating

```bash
node scripts/generate-sdk.mjs           # rewrite sdk/openapi.json
node scripts/generate-sdk.mjs --check   # fail if the committed spec is stale
```

**Route handler docstrings are part of the spec.** FastAPI publishes them as
operation `description` fields, so editing one drifts the committed spec and
trips the gate — correctly, since those descriptions reach generated client
documentation. If CI reports drift on a change that "only touched comments",
this is why: rerun the generator and commit the result.

The spec is read directly off the FastAPI app object — no server, no port, no
database, no lifespan — so the CI gate stays cheap and does not depend on a
healthy runtime.

## Operation IDs are the public method names

Generated clients turn each `operationId` into a method name, which makes those
identifiers part of the external contract. They are derived from **HTTP method
+ path**, never from Python function names:

```
GET    /users/me            ->  get_users_me
POST   /api-keys            ->  post_api_keys
DELETE /api-keys/{key_id}   ->  delete_api_keys_by_key_id
GET    /embed/{token}/data  ->  get_embed_by_token_data
```

A path or method change *is* a breaking API change and should rename the client
method. An internal refactor is not, and cannot. `backend/openapi_ids.py` holds
the rule; `backend/tests/test_openapi_operation_ids.py` pins it.

The trade-off is mechanical, sometimes long names. That is the cost of
decoupling the public surface from internal naming, and it is improvable per
route with an explicit `operation_id=` on the decorator.

## Authentication

Every endpoint takes `Authorization: Bearer <token>`, where the token is either
a JWT from `POST /auth/token` or a `ukip_` API key. One header, two credential
kinds — the backend branches on the `ukip_` prefix.

### API key scopes

Keys carry `read`, `write`, and/or `admin`. The scope a request requires is
derived from the route, not declared per endpoint:

| Request | Required scope |
|---|---|
| `GET`, `HEAD`, `OPTIONS` | `read` |
| `POST`, `PUT`, `PATCH`, `DELETE` | `write` |
| Any method on an administrative surface | `admin` |

Scopes escalate: `admin` implies `write` implies `read`. A handful of `POST`
routes that query rather than mutate (`/rag/query`, `/nlq/query`, `/exports/*`,
…) require only `read`.

**Reading a 403.** If the message names a *scope*, the credential is too narrow
— issue a new key with wider scopes. If it names a *role*, the user lacks
permission and a new key will not help. That distinction is invisible from the
wire, which is exactly why it belongs in SDK documentation.

Enforcement is gated by `UKIP_API_KEY_SCOPES_ENFORCED` on the server. Check
`GET /health` → `features.api_key_scopes_enforced` for a given deployment; while
it is off, scope violations are audited but not blocked.

## Stability

`openapi.json` is generated from the **full** API surface — every route the
application registers. Not all of it is a stability commitment: internal and
admin endpoints appear alongside the ones intended for integrators. Treat the
tagged public surfaces (`entities`, `analytics`, `search`, `ingestion`,
`api-keys`, `widgets`) as the supported contract, and expect the rest to change
without notice.

## Generating the clients

Not yet wired up. The intended toolchain is `@hey-api/openapi-ts` for
TypeScript and `openapi-python-client` for Python, both version-pinned so that
an upstream generator release cannot produce phantom diffs on an unrelated PR.

Blocked on a local toolchain issue rather than a design question: `npx` fails
with `ECOMPROMISED — Lock compromised` on this Windows environment, across both
the shared and an isolated npm cache. Adding the generators as devDependencies
would be the cleaner route, but that regenerates `frontend/package-lock.json`,
which is prohibited on Windows (it strips the Linux native binaries that `sharp`
and friends need). Options, in preference order:

1. Generate on Linux (CI job or WSL) and commit the output.
2. Resolve the Windows npm cache failure, then run locally.
3. Add the generators to `package.json` from a Linux checkout, so the lockfile
   is regenerated on a platform that keeps the native binaries.
